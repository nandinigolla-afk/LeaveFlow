# Deployment Guide

This guide deploys LeaveFlow Pro exactly as specified: **frontend → Vercel, backend → Render, database → Neon PostgreSQL.**

## 1. Database — Neon PostgreSQL

1. Create a free account at [neon.tech](https://neon.tech) and create a new project (e.g. `leaveflow-pro`).
2. Copy the connection string from the Neon dashboard. It looks like:
   ```
   postgresql://<user>:<password>@<host>/<dbname>?sslmode=require
   ```
3. Convert it to a JDBC URL for the backend's `DATABASE_URL` env var:
   ```
   jdbc:postgresql://<host>/<dbname>?sslmode=require
   ```
   (Keep the username/password as separate `DATABASE_USERNAME` / `DATABASE_PASSWORD` env vars — see step 2 below.)
4. No manual schema setup needed — Flyway (`backend/src/main/resources/db/migration/V1__init_schema.sql`) runs automatically on the backend's first boot and creates all tables, indexes, constraints, and the leave-type seed data.
5. **Optional demo data**: to load demo accounts (see [Demo Credentials](../README.md#demo-credentials)), run `database/seed_demo_data.sql` against the Neon database after the first successful deploy:
   ```bash
   psql "postgresql://<user>:<password>@<host>/<dbname>?sslmode=require" -f database/seed_demo_data.sql
   ```

## 2. Backend — Render

1. Push this repository to GitHub.
2. In Render, choose **New → Blueprint** and point it at the repo — it will read `render.yaml` at the repo root and provision the service automatically. (Alternatively: **New → Web Service**, select "Docker" as the runtime, and point `Dockerfile Path` to `backend/Dockerfile` with root directory `backend`.)
3. Set the following environment variables in the Render dashboard (marked `sync: false` in `render.yaml`, so they aren't committed to source control):

   | Key | Value |
   |---|---|
   | `DATABASE_URL` | `jdbc:postgresql://<neon-host>/<dbname>?sslmode=require` |
   | `DATABASE_USERNAME` | from Neon |
   | `DATABASE_PASSWORD` | from Neon |
   | `CORS_ALLOWED_ORIGINS` | your Vercel URL, e.g. `https://leaveflow-pro.vercel.app` |
   | `FRONTEND_URL` | same Vercel URL (used to build password-reset links) |
   | `MAIL_USERNAME` / `MAIL_PASSWORD` | SMTP credentials (e.g. a Gmail App Password) if you want real emails; leave blank to have emails silently no-op |

   `JWT_SECRET` is auto-generated by Render (`generateValue: true`) — don't set it manually.

4. Deploy. Render builds the Docker image (multi-stage Maven build → slim JRE runtime) and exposes the service on the assigned `*.onrender.com` URL. Health checks hit `/actuator/health`.
5. Confirm it's live: `curl https://<your-service>.onrender.com/actuator/health` should return `{"status":"UP"}`.

## 3. Frontend — Vercel

1. In Vercel, **Add New → Project**, import the same GitHub repo, and set **Root Directory** to `frontend`.
2. Vercel will detect the Vite framework automatically (also codified in `frontend/vercel.json`).
3. Set the one required environment variable:

   | Key | Value |
   |---|---|
   | `VITE_API_BASE_URL` | `https://<your-render-service>.onrender.com/api/v1` |

4. Deploy. `frontend/vercel.json` includes an SPA rewrite so client-side routes (e.g. `/employee/history`) don't 404 on refresh.
5. Once deployed, go back to Render and confirm `CORS_ALLOWED_ORIGINS` matches your final Vercel URL exactly (including `https://`, no trailing slash).

## 4. Post-deploy checklist

- [ ] `GET https://<backend>/actuator/health` returns `UP`
- [ ] `https://<backend>/swagger-ui.html` loads and lists all endpoints
- [ ] Frontend loads at the Vercel URL and the login page renders
- [ ] Signup → login → request leave works end-to-end against the deployed backend
- [ ] CORS is locked to your actual frontend origin (not `*`)
- [ ] Demo data seeded (if desired) and demo credentials confirmed working

## Local development (Docker-free)

For local dev without Docker, see the [README](../README.md#local-development-setup) — it covers running Postgres locally (or pointing at Neon directly), `mvn spring-boot:run`, and `npm run dev`.
